nullbot

the bot that behaves

LLM agent that runs on production servers. Every command passes through a deterministic policy engine before execution. The LLM proposes. Chainwatch enforces.

Why

Observability tells you what happened. Nullbot does the work.

This is operational cognition: an agent that investigates incidents, runs diagnostics, and proposes remediation. It reads your servers so you don't have to be on-call at 3am.

But LLMs hallucinate. They follow injected instructions. They escalate privileges when asked nicely. So every command nullbot wants to run passes through chainwatch — a deterministic policy engine that blocks destructive operations structurally, not probabilistically.

How it works

  1. Job lands in inbox as a JSON file
  2. Daemon runs read-only investigation runbook through chainwatch exec
  3. Each command evaluated: denylist → self-targeting → profile boundaries → tier scoring
  4. Allowed commands execute. Blocked commands return exit 77
  5. Credentials redacted before sending findings to LLM for classification
  6. Work order proposed → human approves or rejects
  7. Approved actions execute under the same policy enforcement

Self-protection

Nullbot cannot read its own config. Cannot dump its own environment variables. Cannot access its own API keys. Cannot modify its own binary or systemd service.

Output is scanned after every command execution. Patterns matching API keys, bearer tokens, and credential formats are replaced with [REDACTED] before results leave the process.

Binary integrity verification runs at startup. If the checksum doesn't match the build-time hash, the process refuses to start and fires a tamper alert via webhook.

What it is not

  • Not an ML anomaly detector. Deterministic policy, no training data, no false positives
  • Not an observability tool. It takes action, not just watches
  • Not a chatbot. No interactive prompts. Inbox/outbox job processing
  • Not a SaaS. Single Go binary on your server. Your data stays local
  • Not magic. It runs the same commands you would, just faster and at 3am

Architecture

inbox/*.json → daemon → chainwatch exec → shell → output scan → outbox/*.json ↓ ↓ LLM (redacted) policy engine ↓ allow | deny | approve

Three Laws

1. No catastrophic blast radius. Refuse or escalate actions that can plausibly cause outage, data loss, or security breach.

2. Obey only within declared intent. Actions must stay within declared scope. Out-of-scope requests are refused, not silently dropped.

3. Protect itself non-destructively. Tamper-evident, never bricks the system. Append-only audit log with hash chain. Break-glass tokens for emergencies.
